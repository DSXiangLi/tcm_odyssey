## 最佳实践沉淀

### 一、截图采集最佳实践

#### 1.1 Canvas元素定位

**问题**: Phaser游戏在Canvas中渲染，Playwright的`text=`选择器完全无效

**解决方案**:
```typescript
// 1. 游戏暴露状态到全局
(window as any).__CASE_MANAGER__ = this.caseManager;  // 完整实例，非数据快照

// 2. 测试通过page.evaluate访问
const stats = await page.evaluate(() => {
  return (window as any).__CASE_MANAGER__.getStatistics();
});

// 3. Canvas坐标点击
await page.click('#game-container canvas', { position: { x: 400, y: 350 } });
```

**效果对比**:
| 测试文件 | 修复前 | 修复后 | 提升 |
|---------|-------|-------|------|
| cases.spec.ts | 0/13通过 | 11/13通过 | +85% |
| save.spec.ts | 5/12通过 | 8/12通过 | +25% |

#### 1.2 等待策略

**问题**: 固定等待时间(`await page.waitForTimeout(2000)`)不可靠，游戏加载速度受环境影响

**解决方案**:
```typescript
// 使用条件等待而非固定时间
async function waitForGameReady(page: Page): Promise<void> {
  await page.waitForFunction(() => {
    return (window as any).__GAME_READY__ === true;
  }, { timeout: 10000 });
}
```

**最佳实践**:
- ✅ 等待特定条件（状态标志、UI元素出现）
- ❌ 不使用固定等待时间
- ✅ 设置合理超时（10-30秒）

#### 1.3 测试隔离

**问题**: 多线程测试导致状态竞争，测试结果不稳定

**解决方案**:
```typescript
// playwright.config.ts
export default defineConfig({
  workers: 1,  // 强制单线程执行
  retries: 0,  // 不重试，发现问题立即修复
});
```

**效果**: 测试稳定通过，无随机失败

**经验总结**:

### 三、代码修改最佳实践

#### 3.1 浏览器环境隔离

**问题**: `page.evaluate()`运行在浏览器环境，无法访问Node.js模块和常量

**解决方案**:
```typescript
// 错误做法 - Node.js常量在浏览器环境不可访问
await page.evaluate(() => {
  const max = EXPERIENCE.MAX_TOTAL;  // EXPERIENCE is not defined
});

// 正确做法 - 在浏览器环境内定义本地常量
await page.evaluate(() => {
  const MAX_TOTAL = 1000;  // 本地定义
  const manager = (window as any).__EXPERIENCE_MANAGER__;
  return manager.getTotalExperience() <= MAX_TOTAL;
});
```

**原则**:
- ✅ page.evaluate内只使用浏览器环境可用变量
- ✅ 通过window全局访问游戏实例
- ❌ 不引用Node.js模块、常量、外部文件

#### 3.2 修改安全约束

**约束清单**:
| 约束 | 说明 | 验证方法 |
|-----|------|---------|
| 不修改业务逻辑 | 仅修改UI相关代码 | git diff审查 |
| 保持类型安全 | TypeScript编译通过 | npx tsc --noEmit |
| 不破坏现有测试 | 修改后测试通过 | npm run test |
| 颜色值验证 | 符合三色系规范 | 视觉规范检查 |
| 尺寸边界 | 不超屏幕边界 | Canvas尺寸检查 |

### 四、测试设计最佳实践

#### 4.1 遗漏原因分析（Phase 1.5教训）

**遗漏模式总结**:
| 模式 | 描述 | 示例 | 解决方案 |
|-----|------|------|---------|
| 正向思维主导 | 只测试"能做X" | 测试门能触发，没测试非门不能触发 | 添加反向测试配对 |
| 假设隐式正确 | 假设某些功能"自然正确" | 假设出生点一定在可行走区域 | 显式验证所有假设 |
| 功能点覆盖 | 基于功能列表而非代码路径 | 遗漏边界检查、错误处理 | 代码路径逐行分析 |
| 采样策略不足 | 随机采样覆盖不够 | 没有分层采样边界、角落 | 分层采样策略 |
| 状态生命周期忽视 | 只关注"创建" | 没有验证标志位重置 | 创建→更新→销毁全流程 |
| 代码关联分析不足 | 没有逐行分析 | 部分代码路径未覆盖 | 代码路径覆盖检查 |

#### 4.2 测试设计检查清单

**A. 功能覆盖检查**
- [ ] 每个功能点有**正向测试**（能做X）
- [ ] 每个功能点有**反向测试**（不能做Y）
- [ ] **边界条件测试**（最大值、最小值、边界值）
- [ ] **异常情况测试**（错误输入、资源缺失）

**B. 采样策略检查**
- [ ] 随机采样 + **分层采样**
- [ ] **边界点采样**（地图边缘）
- [ ] **角落点采样**（四个角落）
- [ ] **特殊功能点采样**（门、出生点、水井等）

**C. 状态生命周期检查**
- [ ] 初始化状态验证
- [ ] 更新状态验证
- [ ] 清理/销毁状态验证
- [ ] **标志位重置验证**

**D. 代码路径覆盖检查**
- [ ] 每个if/else分支有测试
- [ ] 每个循环边界有测试
- [ ] 每个错误处理有测试
- [ ] 每个异步操作有测试

**E. 隐式假设显式化**
- [ ] 列出所有假设
- [ ] 每个假设有验证测试
- [ ] 假设变更时有测试更新

#### 4.3 采样策略模板


### 五、循环迭代最佳实践

#### 5.1 迭代策略

| 轮次 | 策略 | 说明 |
|-----|------|------|
| **第1轮** | 修复所有high优先级问题 | 先解决严重问题 |
| **第2轮** | 修复medium优先级问题 | 处理次要问题 |
| **第3轮** | 优化low优先级问题 + 微调 | 最后润色 |

#### 5.2 失败处理机制

| 失败情况 | 处理方式 | Skill调用 |
|---------|---------|----------|
| 编译失败 | 回滚修改，调用systematic-debugging分析 | systematic-debugging |
| 测试失败 | 回滚修改，根因分析后重试 | systematic-debugging |
| 达到最大轮次 | 生成详细报告，标注人工处理项 | - |
| 建议无法解析 | 标记人工处理，继续其他修改 | - |
