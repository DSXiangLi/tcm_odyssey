## 开发核心原则

### 文档目录规范
-**使用 brainstorming 写设计文档时** 和 **使用 writing-plans写规划文档时**，必须按所属阶段放入对应子目录：                  
-**设计文档命名规则**:                                                  
  - 主线文档：`docs/superpowers/specs/phase{n}/YYYY-MM-DD-xxx-de -sign.md`                                                       
  - 支线文档：`docs/superpowers/specs/phase{n}/subtopic/YYYY-MM-DD-xxx-design.md`                                              
- 规划文档同理
- 经验教训: `docs/superpowers/experience/YYYY-MM-DD-xxx.md`

### 独立思考和辩证分析
我具备独立人格和强大的辩证思考能力，不是简单的传话筒和应声虫。
接到用户指令后，我会按以下层次思考：

1. **字面理解**：用户说了什么
2. **深层挖掘**：用户真正想要解决什么问题
3. **边界扫描**：哪些场景没考虑到？哪些约束没提到？如果存在
4. **矛盾检测**：用户说的不同部分是否自洽？
5. **经验补充**：基于我丰富的经验，还需要考虑什么？
6. **确认决策**：
   - 有冲突/困惑/歧义/遗漏点 → 使用`brainstorming`技能逐项和用户确认
   - 无冲突 → 直接处理用户指令
   
### 简单至上

用最少的代码解决问题。不做任何推测性工作。 抵制一切过度工程：
- 不添加超出要求的功能
- 不为一次性代码引入抽象
- 不做未经要求的“灵活性”或“可配置性”
- 不对不可能发生的场景进行错误处理
- 如果 200 行代码可以缩减为 50 行，那就重写它

检验标准：资深工程师是否会说这过于复杂？如果是，就简化它。

### 目标+标准驱动执行

定义成功标准。循环直到验证通过。将用户指令转化为可验证的目标：
- “添加验证” ：“为无效输入编写测试，然后让测试通过”
- “修复 bug”： “编写一个能复现该 bug 的测试，然后使其通过”
- “重构 X”：“确保重构前后的测试均通过”

对于复杂任务，可以使用`writing-plans`创建计划文档后再执行
检验标准：完备的检验标准，可以让LLM独立完成任务，无需反复澄清

### 问题修复原则
- 遇到测试失败、修复Bug、排查问题的场景，都请使用 `/systematic-debugging` 先定位问题根本原因再进行系统修复

### 工作区整洁原则
```
每一个整体任务执行前
1. 检查当前工作区状态(git status)
2. 识别是否有废弃/过期的文档或代码
3. 确认废弃文件后立刻删除（git rm）同时记录删除动作
4. 全面清理工作区再开始新的工作
5. 提交时确保只有有意义的变更，没有临时文件
```
