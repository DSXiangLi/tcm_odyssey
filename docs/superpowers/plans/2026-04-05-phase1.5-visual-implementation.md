## 🆕 新方案：瓦片自动映射系统（2026-04-06）

### 核心思路

**「视觉优先，逻辑后推导」**：
1. 先让AI生成完整小镇全景图（美观优先）
2. 在图片上叠加网格+坐标标注
3. 用AI视觉识别各元素的精确坐标
4. 生成颜色蒙版可视化验证
5. AI验证复核，自动修正问题
6. 输出瓦片配置映射到游戏代码

### 工作流程

```
输入: 小镇全景图（用户已生成 version3）
      │
      ▼

└─────────────────────────────────────────┘
      │
      ▼

│ • 识别街道坐标范围                       │

│ • 识别水域/桥梁坐标                      │
│ • 识别门位置                             │

      │
```

## ⚠️ 重要教训（2026-04-06）

### 之前的错误

1. **只测试图片文件**：对AI生成的素材图片进行视觉验收，而非游戏中实际运行效果
2. **素材与代码不匹配**：AI生成的全景图中建筑位置是随机的，和代码中的碰撞检测、门位置完全对不上
3. **场景切换未验证**：生成的室内素材导致内景无法从游戏界面到达

### 正确的测试方法

```
❌ 错误：对图片文件进行AI视觉分析
✅ 正确：在游戏运行时截图，验证实际显示效果

❌ 错误：AI自由发挥生成全景图
✅ 正确：严格按照map-config.ts中的坐标生成素材

❌ 错误：假设场景切换会自动工作
✅ 正确：验证玩家能从室外进入每个室内场景
```

## 2. 分层架构设计

### 2.1 室外场景分层架构

**设计原则**: 背景层不可交互，建筑层独立可交互，便于扩展和碰撞检测

```
┌─────────────────────────────────────────────────┐
│  UI层        │ 场景名称、提示文字、按钮         │
├─────────────────────────────────────────────────┤
│  玩家层      │ 玩家角色Sprite                  │
├─────────────────────────────────────────────────┤
│  建筑层      │ 诊所、药园、家、池塘、竹林、花田 │
│              │ （独立Sprite，可交互）           │
├─────────────────────────────────────────────────┤
│  背景层      │ 草地、山脉远景、天空             │
│              │ （单张AI生成背景图）             │
└─────────────────────────────────────────────────┘
```

### 2.2 各层详细说明

| 层级 | 内容 | 交互 | 实现方式 | 素材来源 |
|-----|------|-----|---------|---------|
| **背景层** | 草地、山脉远景、天空 | 无 | 单张AI生成背景图 | Seedream生成2048x2048 |
| **建筑层** | 诊所、药园、家 | 进入室内场景 | 独立Sprite | Seedream生成各建筑外观 |
| **交互元素层** | 池塘、竹林、花田 | 采集/钓鱼等 | 独立Sprite | Seedream生成各元素 |
| **玩家层** | 玩家角色 | 移动控制 | 动态Sprite | 待设计角色精灵 |
| **UI层** | 场景名称、提示 | 按钮交互 | Phaser UI | 程序生成 |

### 2.3 建筑层交互定义

| 元素 | 交互类型 | 触发条件 | 效果 |
|-----|---------|---------|------|
| **诊所** | 场景切换 | 玩家接近按空格 | 进入ClinicScene |
| **药园** | 场景切换 | 玩家接近按空格 | 进入GardenScene |
| **家** | 场景切换 | 玩家接近按空格 | 进入HomeScene |
| **池塘** | 采集交互 | 玩家接近按空格 | 触发钓鱼小游戏/获取水资源 |
| **竹林** | 采集交互 | 玩家接近按空格 | 采集竹笋道具 |
| **花田** | 采集交互 | 玩家接近按空格 | 采集花卉/药材 |

### 2.4 分层架构优势

1. **背景层不需要切割** - 直接用AI生成整体场景图，节省瓦片制作工作量
2. **建筑/交互元素独立** - 碰撞检测和交互事件更精确，每个元素可单独定义交互范围
3. **后续扩展容易** - 添加新建筑只需新增Sprite，不影响背景层
4. **性能优化** - 背景层静态图片可预渲染，建筑层按需加载

## 6. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|-----|------|---------|
| AI生成风格不统一 | 视觉不协调 | 使用统一Prompt模板，固定seed值 |
| 中医素材不准确 | 教学价值受损 | AI生成后人工审核，必要时调整Prompt |
| 分层架构性能问题 | 游戏卡顿 | 背景层预渲染，建筑层按需加载 |
| 用户不满意氛围 | 需要重新设计 | 在素材生成阶段充分验证风格 |
| 建筑层碰撞检测不准 | 交互体验差 | 精确定义每个建筑的交互区域 |
