## 1. 设计概述

### 1.1 测试定位

Phase 1.5相比Phase 1有重大架构变化：

| 对比项 | Phase 1 | Phase 1.5 |
|-------|---------|-----------|
| 地图尺寸 | 40×30瓦片 | 86×48瓦片 |
| 碰撞检测 | 瓦片类型判断 | 可行走瓦片集合 |
| 背景渲染 | 占位瓦片 | AI全景图 |
| 门坐标 | (7,8), (33,8), (6,26) | (15,8), (60,8), (61,35) |
| 出生点 | (20,15) | (47,24) |

**测试范围**: 功能性验证 + 视觉验证 + 性能验证 + 用户体验验证

### 1.2 核心验收目标

**Phase 1.5完成标志**:
> 玩家可以真正"走进完整的游戏世界"，感受到田园治愈+中医文化+神秘探索的综合氛围

**验收维度**:
| 维度 | 权重 | 验证方式 |
|------|------|---------|
| **功能性** | 40% | 自动化测试 + BFS算法验证 |
| **视觉质量** | 30% | AI多模态分析 |
| **性能** | 15% | 性能监控工具 |
| **用户体验** | 15% | 用户问卷 |

### 1.3 核心设计原则

| 原则 | 描述 |
|------|------|
| **分层瀑布式执行** | 每层是下一层的前置条件，快速失败，节省时间 |
| **最大自动化** | 尽一切可能使用自动化测试，把人工投入降到最低 |
| **AI深度验证** | 不仅验证加载正确性，还分析视觉质量、氛围、可辨识度 |
| **全覆盖验证** | 验证所有可行走区域连通、所有门可达、所有不可行走区域不可达 |
| **基于证据修复** | 失败时不立即修复，先记录日志、分析原因、定位根因，再修复 |

## 4. AI视觉深度分析策略
### 4.1 分析工具选择

| 分析类型 | 工具/API | 用途 |
|---------|---------|------|
| 截图质量检查 | QWEN VL / GLM-4V | 验证背景图正确加载、无破损 |
| 建筑区域识别 | QWEN VL | 识别截图中建筑位置和数量 |
| 门区域辨识 | QWEN VL | 验证门区域在视觉上可辨认 |
| 氛围评分 | QWEN VL + GLM-4 | 综合评分和建议生成 |

### 4.2 截图采集策略

```
截图类型:
├── 全景截图 (2752×1536) - 用于整体分析
├── 出生点截图 (320×320) - 验证出生点周围环境
├── 药园门截图 (320×320) - 验证药园门区域
├── 诊所门截图 (320×320) - 验证诊所门区域
├── 家门截图 (320×320) - 验证家门区域
└── 随机采样截图 (320×320 × 10张) - 验证各区域质量
```

### 4.3 AI分析Prompt模板

**背景图质量检查Prompt**:
```
请分析这张游戏截图，回答以下问题：
1. 图片是否正常显示？（无黑屏、无破损、无异常色块）
2. 能否识别出游戏场景？（是/否，简述理由）
3. 图片清晰度如何？（1-5分）
请以JSON格式返回结果。
```

**建筑区域识别Prompt**:
```
请分析这张游戏截图，识别其中的建筑：
1. 能否看到建筑？有几个？
2. 建筑位置分别在图片的哪个区域？（左上/右上/左下/右下/中间）
3. 建筑风格是否与周围环境协调？
请以JSON格式返回结果。
```

**氛围评分Prompt**:
```
作为游戏视觉设计师，请为这张游戏截图评分（1-5分）：
1. 田园治愈感：画面是否传达温馨、放松的氛围？
2. 中医文化感：是否有中医元素、古朴风格？
3. 探索引导感：是否有引导玩家探索的视觉暗示？
4. 整体和谐度：所有元素是否协调统一？
请给出每项评分和简要理由，以及总体评价。
```

### 4.4 通过标准
| 分析项 | 通过条件 | 置信度要求 |
|--------|---------|-----------|
| 背景图质量 | 无异常 + 清晰度≥3分 | ≥80% |
| 建筑区域识别 | 正确识别≥2个建筑 | ≥80% |
| 门区域可辨识 | 门区域清晰可辨 | ≥80% |
| 氛围评分 | 加权平均≥3.5/5 | ≥80% |

## 5. 分层执行流程
### 5.1 执行流程图
```
执行命令: npm run test:phase1-5

┌─────────────────────────────────────────────────────────┐
│  Layer 1: 冒烟测试 (必须100%通过)                    │
│  └── 任一测试失败 → 停止执行，输出错误报告            │
├─────────────────────────────────────────────────────────┤
│  Layer 2: 功能性测试 (必须≥90%通过)                  │
│  └── 通过率<90% → 停止执行，记录失败项                │
├─────────────────────────────────────────────────────────┤
│  Layer 3: 深度验证 (评分制)                              │
│  └── 记录各项评分，不阻断执行                          │
├─────────────────────────────────────────────────────────┤
│  Layer 4: 性能测试 (阈值判定)                         │
│  └── 记录性能数据，不阻断执行                          │
├─────────────────────────────────────────────────────────┤
│  Layer 5: 用户验收 (人工执行)                         │
│  └── 生成问卷链接，等待用户反馈                      │
└─────────────────────────────────────────────────────────┘
                          ↓
                    生成汇总报告
```

### 5.2 通过标准汇总
| Layer | 通过条件 | 阻断执行 |
|-------|---------|---------|
| Layer 1 | 100%通过 | 是 |
| Layer 2 | ≥90%通过 | 是（低于90%停止） |
| Layer 3 | 加权分≥3.5/5 | 否（记录但不阻断） |
| Layer 4 | 全部在阈值内 | 否（记录但不阻断） |
| Layer 5 | 平均分≥3.5/5 | 否（人工验收） |

### 5.3 失败处理原则
1. **不立即修复**：失败时先记录日志和截图
2. **补充日志**：在修复前先补充必要的日志输出
3. **充分分析**：分析错误信息、状态数据、截图
4. **定位原因**：确定是代码问题、配置问题还是资源问题
5. **再修复**：确认原因后再进行修复
6. **禁止猜测**：不基于猜测进行修复，必须有充分证据
7. **坚韧不拔**：遇到问题有20次自己尝试解决的机会，超过20次再询问用户

## 6. 核心工具接口设计
### 6.1 可行走区域验证器 (walkable-verifier.ts)
```typescript
interface WalkableVerifier {
  // 验证单个位置是否可行走
  isWalkable(tileX: number, tileY: number): boolean;

  // 验证位置是否可到达（BFS）
  canReach(from: TilePos, to: TilePos): boolean;

  // 获取从A到B的路径
  findPath(from: TilePos, to: TilePos): TilePos[] | null;

  // 验证所有可行走区域连通性
  verifyAllConnected(): { connected: boolean; isolatedRegions: TilePos[][] };

  // 随机采样可行走区域
  sampleWalkableTiles(count: number): TilePos[];

  // 验证门周围有可行走区域
  verifyDoorAccessibility(doorPos: TilePos): boolean;
}
```

### 6.2 AI视觉分析器 (visual-analyzer.ts)
```typescript
interface VisualAnalyzer {
  // 分析背景图质量
  analyzeBackgroundQuality(screenshot: Buffer): Promise<QualityResult>;

  // 识别建筑区域
  identifyBuildings(screenshot: Buffer): Promise<BuildingResult>;

  // 分析门区域可辨识度
  analyzeDoorClarity(screenshot: Buffer, doorPos: TilePos): Promise<DoorResult>;

  // 氛围评分
  rateAtmosphere(screenshot: Buffer): Promise<AtmosphereScore>;
}
```
