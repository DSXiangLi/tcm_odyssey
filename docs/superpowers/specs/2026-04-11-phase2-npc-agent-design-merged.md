## 1. 设计哲学

### 1.1 核心认知转变

| 传统游戏NPC | AI Agent NPC |
|------------|-------------|
| 被动执行预设逻辑的木偶 | 有自主意识的数字生命 |
| 玩家触发 → NPC响应 | NPC有目标、日程、情感 |
| 静态对话树 | 动态生成教学内容 |
| 任务是数据 | 任务是教学手段 |
| 记忆是变量 | 记忆是演变的认知 |

### 1.2 "有灵魂的NPC"定义

一个真正有灵魂的NPC应该具备：

| 维度 | 定义 | Hermes对应 |
|-----|------|-----------|
| **身份认同** | 我是谁？我的专业？我的性格？ | SOUL.md |
| **记忆系统** | 我对这个学生了解什么？ | USER.md |
| **教学大纲** | 我要教什么？教学计划是什么？ | SYLLABUS.md (Skill) |
| **任务管理** | 学生学到哪了？下一步学什么？ | TASKS.json + Tools |
| **自主行动** | 我能主动发起什么？ | 登录触发 + Tools |

## 2. NPC 角色设计

### 2.1 多导师体系

玩家是唯一的学生，但有多个老师，每个老师负责不同领域：

| NPC | 专长领域 | 教学内容 |
|-----|---------|---------|
| **青木先生** | 伤寒论 | 解表药 + 外感类疾病 |
| **老张** | 药材种植 | 药材辨识、种植、炮制 |
| **（未来）李大夫** | 金匮要略 | 杂病诊治 |
| **（未来）王药师** | 本草学 | 药材鉴别、产地认知 |

### 2.2 青木先生详细设定

**身份**：百草镇唯一的中医师，三代行医

**专长**：伤寒论，外感病诊治

**教学内容**：解表类方剂（桂枝汤、麻黄汤、银翘散、桑菊饮）

**教学大纲范围**：
- 方剂组成与功效
- 辨证论治（风寒表实 vs 表虚 vs 风热）
- 选方用药

## 4. 教学大纲设计 (SYLLABUS.md)

### 4.1 大纲结构原则

**主线选择**：以**中医内科学（病症）**为主线，方剂为二级目录

**层级结构**：
```
病症（外感病）→ 病类（风寒/风热）→ 证型（表实/表虚/犯卫）→ 方剂（麻黄汤/桂枝汤/银翘散）
```

### 4.2 大纲核心内容（摘要）

**第一篇：外感病总论**
- 外感病概念、病因（六淫）
- 辨证体系对比（六经辨证 vs 卫气营血辨证 vs 三焦辨证）
- 学派归属（伤寒学派 vs 温病学派）
- 治疗原则（辛温解表 vs 辛凉解表）

**第二篇：风寒外感（伤寒）**
- 证型一：风寒表实证（麻黄汤证）
  - 病机：风寒束表，营卫郁滞
  - 核心：无汗（关键鉴别）、脉浮紧
  - 方剂：麻黄汤（麻黄、桂枝、杏仁、甘草）

- 证型二：风寒表虚证（桂枝汤证）
  - 病机：风邪袭表，营卫不和
  - 核心：有汗（关键鉴别）、脉浮缓
  - 方剂：桂枝汤（桂枝、芍药、甘草、生姜、大枣）

**第三篇：风热外感（温病）**
- 证型三：风热犯卫证（银翘散证）
  - 核心：咽痛明显、发热重
  - 方剂：银翘散

- 证型四：风温咳嗽证（桑菊饮证）
  - 核心：咳嗽为主
  - 方剂：桑菊饮

**第四篇：综合对比与鉴别**
- 四方综合鉴别表
- 辨证思路流程图
- 时令季节用药参考

## 5. 任务系统设计 (TASKS.json)

### 5.1 两层结构：Task → Todo

**设计理念**：
- **Task**：对应方剂/内科证型层级，玩家看到的主要单位
- **Todo**：每个Task的具体知识维度细分，用于精确追踪

```
Task（任务）- 方剂/内科证型层级
例："麻黄汤学习"、"风寒表实证辨证"
- 有整体进度(0-100%)
- 当所有Todo完成时，Task自动完成
         │ 包含多个
         ▼
Todo（待办）- 知识维度细分
方剂类：组成、配伍、煎服法、禁忌、鉴别、方歌
内科类：病机、辨证要点、脉诊、舌诊、选方
- 有独立掌握度(0-1)
- mastery ≥ 0.7 视为完成
```

### 5.3 Todo维度模板

**方剂类Task的Todo模板**：

| Todo ID | 名称 | 内容 | 游戏形式 |
|---------|------|------|---------|
| composition | 组成 | 君臣佐使、用量比例 | 煎药游戏 |
| compatibility | 配伍 | 药对配伍意义 | 煎药游戏+对话 |
| effect | 功效 | 功效主治 | 选择题 |
| decoction | 煎服法 | 特殊煎煮服用方法 | 对话+游戏 |
| contraindication | 禁忌 | 使用禁忌 | 选择题 |
| differentiation | 鉴别 | 类方鉴别要点 | 对比选择+论述 |
| formula | 方歌 | 记忆口诀 | 填空题 |

**内科证型类Task的Todo模板**：

| Todo ID | 名称 | 内容 | 游戏形式 |
|---------|------|------|---------|
| pathology | 病机 | 病因病机分析 | 对话讲解 |
| key_points | 辨证要点 | 核心症状、鉴别要点 | 选择题 |
| pulse | 脉诊 | 脉象古文理解 | 诊治游戏 |
| tongue | 舌诊 | 舌象特征观察 | 诊治游戏 |
| prescription | 选方 | 证型对应方剂 | 诊治游戏 |

### 5.4 任务解锁机制

| 触发类型 | 说明 | 示例 |
|---------|------|------|
| **大纲驱动** | 前置Task完成后自动解锁 | 麻黄汤学习完成 → 麻黄汤辨证解锁 |
| **blocked_by字段** | Task/病案通过blocked_by字段声明依赖 | blocked_by: "mahuang-tang-learning" |
| **经验值解锁** | 功能解锁通过经验值阈值 | 方剂加减功能需经验值达到X |
| **玩家请求** | 玩家主动询问学习 | NPC推荐下一个Task |

**核心原则：任务是游戏操作，而非对话**

### 6.1 任务类型与游戏映射

### 7.2 问诊环节设计（重构）

#### 7.2.1 设计理念

**从固定6问 → AI病人 + 自由追问**

- 病人是AI NPC，有性格、说话风格
- 玩家自由追问，系统后台追踪关键信息
- 解密游戏感：玩家从描述中"捕捉"线索

#### 7.2.2 问诊流程

```
病人（AI NPC）进门 → AI自主描述主诉 → 玩家自由追问 → AI实时回答（流式输出）
                                    ↓
                            后台AI分析线索获取状态
                                    ↓
                            必须线索收集齐 → 玩家确认 → 进入切脉环节
```

#### 7.2.5 线索追踪机制

**线索层级**：

| 层级 | 定义 | 作用 |
|-----|------|------|
| **必须线索** | 辨证必需的关键信息 | 收集齐后才能进入下一环节 |
| **辅助线索** | 加深理解、增强诊断准确性 | 不强制，收集可获得经验值 |

**AI语义分析**（每轮追问后）：

```python
@tool
def analyze_clue_collection(
    dialogue_history: list,
    clue_definitions: dict
) -> dict:
    """
    分析对话记录，判断关键线索是否已获取

    Returns:
        {
            "恶寒重": True,
            "无汗": True,
            "发热程度": False,
            "身痛": True,
            ...
        }
    """
    pass
```

**问诊结束条件**：
1. 所有必须线索收集完成
2. 弹出确认提示，玩家可选择继续追问辅助线索或进入下一环节

### 15.3 通信协议

**HTTP + SSE流式输出**：

```typescript
async function chatStream(
  context: object,
  userMessage: string,
  onChunk: (text: string) => void
): Promise<void> {
  const response = await fetch('http://localhost:8642/v1/chat/stream', {
    method: 'POST',
    body: JSON.stringify({ context, user_message: userMessage })
  });

  // SSE处理
  const reader = response.body.getReader();
  while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    onChunk(new TextDecoder().decode(value));
  }
}
```
