## 1. 设计概述

### 1.1 测试定位

Phase 1 AI端到端测试聚焦于 **功能性验证**，而非视觉风格验收（视觉风格验收在Phase 1.5执行）。

**验证范围**：
- 布局功能性验证（地图尺寸、路径位置、建筑入口、边界墙壁）
- 移动逻辑验证（响应、速度标准化、碰撞阻断）
- 场景切换验证（门交互、出生点、防抖）
- 室内功能验证（尺寸、功能区域存在性）

**不验证范围**：
- 视觉风格美观度
- 像素艺术质量
- 色彩搭配
- UI细节精致度

### 1.2 核心设计原则

| 原则 | 描述 |
|------|------|
| **最大自动化** | 尽一切可能使用AI测试，把人工投入降到最低 |
| **人工验收精度** | 验收标准参照人工验收的最高标准，确保精度 |
| **多维验证** | 截图分析 + 状态数据 + 游戏日志 + AI Feedback |
| **分级判定** | 硬性标准精确匹配、软性标准容差匹配、视觉判断置信度阈值 |

### 2.2 日志系统架构

采用 **事件总线 + 独立Logger监听** 架构：

```
┌─────────────────────────────────────────────────────────────┐
│                    游戏逻辑层                                │
│  Player / Scenes / SceneManager                             │
│     ↓ 发送事件                                               │
├─────────────────────────────────────────────────────────────┤
│                    EventBus（事件总线）                       │
│  - player:move, player:stop, player:collide                 │
│  - scene:create, scene:switch, scene:destroy                │
│  - door:enter, door:exit                                    │
│  - error:*                                                  │
│     ↓ 多个监听器订阅                                         │
├─────────────────────────────────────────────────────────────┤
│                    GameLogger（日志收集器）                   │
│  - 监听所有事件并记录                                        │
│  - 按类别分文件导出                                          │
│     ↓                                                       │
├─────────────────────────────────────────────────────────────┤
│                    测试系统                                  │
│  - AI端到端测试读取日志                                      │
│  - 截图分析 + 状态提取                                       │
│  - AI Feedback生成报告                                       │
└─────────────────────────────────────────────────────────────┘
```

**事件定义**：

```typescript
// src/systems/EventBus.ts
export const GameEvents = {
  // 玩家事件
  PLAYER_MOVE: 'player:move',
  PLAYER_STOP: 'player:stop',
  PLAYER_COLLIDE: 'player:collide',
  PLAYER_POSITION: 'player:position',

  // 场景事件
  SCENE_CREATE: 'scene:create',
  SCENE_SWITCH: 'scene:switch',
  SCENE_DESTROY: 'scene:destroy',
  SCENE_READY: 'scene:ready',

  // 交互事件
  DOOR_INTERACT: 'door:interact',
  DOOR_ENTER: 'door:enter',
  DOOR_EXIT: 'door:exit',

  // 错误事件
  ERROR: 'error',
  ERROR_INPUT: 'error:input',
  ERROR_COLLISION: 'error:collision'
} as const;
```

**日志类别**：

| 类别 | 文件名 | 内容 |
|------|--------|------|
| scene | scene-{timestamp}.log | 场景生命周期、切换记录 |
| player | player-{timestamp}.log | 移动、碰撞、位置变化 |
| interaction | interaction-{timestamp}.log | 门交互、按键事件 |
| error | error-{timestamp}.log | 异常、警告、失败操作 |

### 4.2 判定标准说明

| 判定类型 | 定义 | 通过条件 |
|---------|------|---------|
| **硬性精确** | 设计文档明确数值 | 必须100%匹配 |
| **状态精确** | 内部状态数据验证 | 状态值必须等于设计值 |
| **视觉阈值** | AI截图分析判断 | 置信度≥80%视为通过 |
| **日志验证** | 事件日志记录检查 | 日志中存在预期事件序列 |
| **动作序列** | 录制帧分析 | 动作连贯，无异常中断 |

## 6. AI Feedback输出

### 6.1 AI Feedback功能范围

| 功能 | 描述 | 输出位置 |
|------|------|---------|
| **结果解读** | 分析测试数据，给出通过/不通过判断 | 报告正文 |
| **诊断建议** | 测试失败时提供可能的修复建议 | 报告"诊断建议"章节 |
| **报告生成** | 自动生成人类可读的测试报告摘要 | Markdown报告 |
| **改进建议** | 基于测试结果提出游戏体验改进建议 | 报告"改进建议"章节 |

### 6.2 报告格式

**JSON报告结构**：

```json
{
  "phase": "Phase1",
  "timestamp": "2026-04-05T...",
  "execution_time_ms": 45000,
  "result": "pass",
  "tests": [
    {
      "id": "T-V001",
      "name": "地图尺寸40x30",
      "status": "pass",
      "verification": {
        "expected": { "width": 40, "height": 30 },
        "actual": { "width": 40, "height": 30 },
        "match": true
      },
      "ai_confidence": 1.0
    }
  ],
  "ai_feedback": {
    "summary": "...",
    "diagnosis": [...],
    "improvements": [...]
  },
  "screenshots": [...],
  "logs": [...]
}
```
